SMART COMMERCE AR - OMNICANALIDAD

Memoria Técnica:
Proyecto PIZHUN

Desarrollo de un prototipo funcional que digitaliza la experiencia de compra. El cliente abre la aplicación, interactúa con un catálogo virtual, se prueba virtualmente la prenda mediante Face Tracking AR y accede directamente a la pasarela web de pago con un solo toque.

01. Estrategia de Ecommerce (Tema 14)

PIZHUN es una marca ficticia de Streetwear Tecnológico. Nuestro producto principal es la PIZHUN Neural Cap (Gorra Táctica). El objetivo es que el cliente experimente el producto mediante un Virtual Try-On (probador virtual inmersivo) desde su propio dispositivo y cierre la transacción en nuestro entorno web seguro.

1.1

Estrategia de Precios (Pricing)

Para definir la viabilidad del producto en el mercado digital, se ha establecido la siguiente estructura de costes e impuestos basándonos en los requisitos del Tema 14:

25.00€
Coste Producción
40%
Margen de Beneficio
21%
IVA Aplicado
42.35€
Precio Final (PVP)

Justificación Estratégica: El coste de fabricación (materiales impermeables) es de 25€. Aplicando un margen de beneficio del 40% (10€ de ganancia neta), obtenemos una base imponible de 35€. Al sumar el 21% de IVA legal en España (7.35€), el precio final al consumidor es de 42.35€. Este precio es altamente competitivo en el nicho premium, absorbiendo los costes de la plataforma AR.

1.2

La Web como Capa de Confianza y Transacción

Mientras que la aplicación de Unity genera el efecto "wow" del probador, la landing page web es la encargada de cerrar la venta. Se ha diseñado siguiendo una estricta filosofía Mobile-First:

  • Header & Hero: Logotipo visible, carrito y un CTA claro invitando a la compra tras la experiencia AR.
  • Ficha Técnica: Detalles "fríos" que no caben en la AR (composición de materiales, impermeabilidad, tallaje).
  • Validación Legal: El Footer incluye políticas de devolución y métodos de pago seguros, elementos vitales para transmitir seguridad en un ecommerce moderno.

02. Generación 2D (Imágenes y Vídeos) con Grok AI

Para cumplir con el criterio de "Calidad de Activos", utilizamos Grok Imagine de forma exclusiva para la generación visual 2D. Esto abarca las fotografías planas del producto base y los vídeos de fondo para la página web.

Prompt de Imagen (Fotografía Base 2D del Producto):

"A hyper-realistic frontal product shot of a futuristic tech-core streetwear cap, matte black waterproof fabric, tactical buckles, volumetric studio lighting, 8K resolution, clean product photography."
⚠ Alucinaciones en la IA de Vídeo
Obstáculo: Grok deformaba las prendas al generar el video de fondo web (formato .mp4) si le pedía "movimiento".
Solución: Modifiqué el prompt añadiendo "perfectly static subject" y "handheld camera movement". Al aislar la física y pedir que solo se moviese la cámara virtual, la imagen estática de la ropa mantuvo su coherencia estética en el vídeo generado.

03. Proceso Image-to-3D con Meshy

Meshy.ai fue el encargado de la conversión espacial. Tomó las imágenes planas generadas en Grok y aplicó el proceso Image-to-3D para generar los modelos volumétricos finales para Unity.

  1. Subimos la fotografía frontal (imagen 2D) generada en el paso anterior.
  2. Configuramos la exportación a High Quality (PBR Textures). Las texturas PBR son vitales para que la malla reaccione a la luz del mundo real en la experiencia de Realidad Aumentada.
  3. Exportamos el modelo tridimensional en formato .glb, el más optimizado para la memoria RAM de los dispositivos móviles.
  4. Ajuste de Escala Crítico: Los modelos 3D generados por IA cargan con un tamaño colosal. En Unity seleccionamos este Prefab y le aplicamos una escala matemática de 0.05 en los ejes X,Y,Z para que la prenda tenga un tamaño realista anatómico y encaje con la cabeza del usuario.

05. Desarrollo Técnico AR y UI (Tema 13: Unity)

En este proyecto, se ha evolucionado la rúbrica base para crear un producto comercial realista. En lugar de forzar al usuario a imprimir un marcador físico estático, se ha implementado Face Tracking AR (Probador Virtual) y una arquitectura de interfaces gráficas completas (Canvas UI).

Paso 1: Arquitectura de Menús (UI/Canvas Flow)

Para que la aplicación funcione como un catálogo real, se desarrolló el siguiente flujo de pantallas interactivo:

  1. Menú de Productos: Al abrir la App, el usuario ve un Canvas con la lista de productos disponibles en la colección.
  2. Pop-up de Detalle: Al seleccionar una prenda, se activa un panel con la información del artículo y un botón para iniciar la experiencia.
  3. Escena AR (Overlay): Al entrar al modo AR, encima de la cámara del usuario se superpone un Canvas Overlay que contiene dos botones fijos: "Volver al Menú" y "Comprar en Web".
Mejora UX/UI frente a la Rúbrica Base
Evolución del Tracking: Dado que vendemos una gorra, usar "Image Tracking" (escanear una foto en la mesa para ver la gorra flotar) no aporta valor al cliente. Decidimos implementar AR Face Manager para anclar la prenda a la anatomía del usuario, creando un Probador Virtual (Try-On) realista.

Evolución de Interfaz: La rúbrica sugería colocar un BoxCollider en el modelo 3D y usar OnMouseDown para ir a la web al tocarlo. Dado que la gorra se mueve dinámicamente con la cabeza del usuario, obligarlo a intentar atinarle con el dedo en la pantalla provoca "fricción" y mala experiencia de usuario. Por ello, prescindimos de los colliders y programamos Botones de Interfaz estáticos (Canvas) para garantizar la accesibilidad y un salto cómodo hacia la web.

Paso 2 y 3: Configuración AR Foundation (Face Tracking)

  1. Instalación: En Window > Package Manager, instalamos AR Foundation y el ARCore XR Plugin.
  2. Setup de Escena (Face Tracking): Eliminamos la cámara tradicional e introdujimos el AR Session Origin. En el Inspector, añadimos el script AR Face Manager en sustitución del rastreador de imágenes.
  3. Asignación: A la propiedad *Face Prefab* de este Manager se le asigna nuestro modelo de gorra. El motor detecta la malla facial a través de la cámara frontal y superpone la prenda automáticamente.

Paso 4: Programación de Controladores (C#)

Para lograr la interacción fluida entre la cámara, los modelos y la pasarela web, se desarrollaron scripts avanzados depurados con IA, comentados de forma exhaustiva para esta memoria.

Prompt de Programación (ChatGPT):

Comenta cada línea detalladamente."

1. ARHatController.cs (Solución al Offset Anatómico)

// Librerías necesarias para usar las físicas y la Realidad Aumentada de Unity using UnityEngine; using UnityEngine.XR.ARFoundation; // Exige al motor que el objeto que lleve este script tenga un componente ARFace adosado [RequireComponent(typeof(ARFace))] public class ARHatController : MonoBehaviour { private ARFace _arFace; private GameObject _currentHatInstance; // Inicialización de los enlaces del componente void Awake() { _arFace = GetComponent<ARFace>(); } void OnEnable() { _arFace.updated += OnFaceUpdated; } void OnDisable() { _arFace.updated -= OnFaceUpdated; } // Función que se ejecuta en bucle cada vez que el sensor del móvil actualiza la posición del rostro private void OnFaceUpdated(ARFaceUpdatedEventArgs args) { if (_currentHatInstance != null) { // Si la cámara está rastreando activamente una cara, activamos el render del modelo if (_arFace.trackingState == UnityEngine.XR.ARSubsystems.TrackingState.Tracking) _currentHatInstance.SetActive(true); else _currentHatInstance.SetActive(false); } } public void SetHatPrefab(GameObject newHatPrefab) { if (_currentHatInstance != null) Destroy(_currentHatInstance); if (newHatPrefab != null) { // Instancia en memoria la gorra (Prefab 3D) _currentHatInstance = Instantiate(newHatPrefab); _currentHatInstance.transform.SetParent(transform, false); _currentHatInstance.transform.localScale = Vector3.one; // EL GRAN ARREGLO ESPACIAL DE UX: // Por defecto, Unity sitúa el ancla ARFace en el centro interno de la cabeza del usuario. // Este 'offset' desplaza la gorra 15cm hacia arriba (eje Y) y 5cm hacia adelante (eje Z), // posando la prenda exactamente sobre la frente real del cliente. _currentHatInstance.transform.localPosition = new Vector3(0, 0.15f, 0.05f); // Bloquea la rotación inicial para evitar deformaciones matemáticas (Quaternions) _currentHatInstance.transform.localRotation = Quaternion.identity; } } }

3. HologramGlitchAnimator.cs (Estética UI en Menús)

// Gestiona los efectos de parpadeo simulando un sistema antiguo al abrir un popup. using System.Collections; using UnityEngine; using UnityEngine.UI; [RequireComponent(typeof(CanvasGroup))] public class HologramGlitchAnimator : MonoBehaviour { private CanvasGroup _canvasGroup; private RectTransform _rectTransform; public float transitionDuration = 0.4f; public float startScaleY = 0.05f; private Vector3 _originalScale; private void Awake() { _canvasGroup = GetComponent<CanvasGroup>(); _rectTransform = GetComponent<RectTransform>(); _originalScale = Vector3.one; } private void OnEnable() { // Inicia la rutina de animación asíncrona StartCoroutine(GlitchInCoroutine()); } private IEnumerator GlitchInCoroutine() { float elapsed = 0f; _canvasGroup.alpha = 0.2f; // Bucle que altera la opacidad y vibración (jitter) durante 0.4 segundos while (elapsed < transitionDuration) { elapsed += Time.unscaledDeltaTime; float t = elapsed / transitionDuration; if (Random.value > t) _canvasGroup.alpha = Random.Range(0.1f, 0.8f); else _canvasGroup.alpha = Mathf.Lerp(0.5f, 1f, t); yield return null; } // Restaura los valores de escala y opacidad puros al terminar _canvasGroup.alpha = 1f; _rectTransform.localScale = _originalScale; } }

Paso 5: Compilación y Exportación (Android Build)

  1. Ajustes de Rendering: En Project Settings > Player, forzamos el uso de OpenGLES3 y eliminamos Vulkan (API que rompe el feed de cámara en la mayoría de Androids).
  2. Build Settings: En File > Build Settings seleccionamos Android y pulsamos Switch Platform.
  3. Incluimos las escenas del proyecto (MenuPrincipal y EscenaAR) usando el botón Add Open Scenes.
  4. Pulsamos Build, obteniendo el ejecutable .apk final para instalar en el móvil y grabar la demostración obligatoria de la entrega.
